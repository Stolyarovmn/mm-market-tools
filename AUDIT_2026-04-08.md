# Audit

Дата: 2026-04-08

## Главный вывод

Проект уже перестал быть просто набором черновых скриптов и стал полезным исследовательским тулкитом. Но в текущем виде он всё ещё собран как `итеративный research stack`, а не как целостная аналитическая система.

Если цель проекта:

- повышать продажи,
- повышать прибыль,
- и позже перейти в браузерный интерактивный режим,

то проект не нужно выбрасывать и пересобирать с нуля. Его нужно `собрать вокруг единой модели данных и слоёв аналитики`.

## Что уже хорошо и не нужно ломать

- Есть рабочий слой `market/category intelligence`:
  - продавцы категории,
  - поиск похожих товаров,
  - сравнение с конкурентами.
- Есть рабочий слой `content analysis`:
  - сравнение карточек,
  - группировка по идеям,
  - план роста.
- Появился правильный сдвиг в методологии:
  - отделение исторического слоя от оконного слоя.
- Есть уже сохранённые артефакты, с которых можно строить повторяемый процесс.

Это не стоит переписывать с нуля.

## Основные проблемы проекта

### 1. Нет единой модели данных

Сейчас проект оперирует множеством форматов:

- seller API JSON
- category crawl JSON
- reports CSV
- ad hoc JSON summaries

Но нет единой нормализованной сущности вида:

- `product`
- `shop`
- `group`
- `period_metrics`
- `inventory_metrics`
- `competitor_metrics`

Из-за этого логика дублируется и отчёты трудно комбинировать.

### 2. Слишком много логики в отдельных скриптах

Во многих скриптах повторяются:

- HTTP helpers
- CSV parsing
- tokenization / similarity logic
- report writing
- dated output naming

Это признак, что проекту нужен общий `core` слой.

### 3. Ранние отчёты содержат исторические методологические ошибки

Самая важная:

- накопительный `quantitySold` использовался как будто это текущий спрос.

Теперь это исправлено концептуально, но старые отчёты нужно считать `historical / exploratory`, а не операционными.

### 4. Нет тестов и контрольных примеров

Сейчас корректность проверяется в основном руками:

- запустился ли CLI;
- создался ли файл;
- выглядит ли результат разумно.

Для проекта такого типа нужны хотя бы:

- фиксированные sample CSV / JSON fixtures;
- smoke tests для парсеров;
- ожидания по ключевым столбцам и полям.

### 5. Нет слоя интерактивного использования

Сейчас всё строится вокруг CLI и markdown/csv/json файлов.

Это нормально на исследовательском этапе, но для регулярного использования в браузере нужен отдельный presentation слой.

## Что нужно добавить

### 1. Нормализованный data layer

Минимум:

- `data/raw/`
- `data/normalized/`
- `reports/`

И функции нормализации:

- official sells report -> normalized sales rows
- left-out report -> normalized inventory rows
- product groups -> normalized idea rows

### 2. Metrics layer

Нужен единый расчёт метрик:

- sales velocity
- stock cover
- sell-through
- ABC / XYZ
- gross profit
- margin
- group revenue / group profit

### 3. Decision layer

Отдельные отчёты для действий:

- what to reorder
- what to restock urgently
- what to discount
- what to discontinue
- what idea to expand
- what competitor signal to follow

### 4. UI-ready outputs

Для будущего браузерного интерфейса нужно готовить не только markdown, но и компактные JSON-слои для виджетов:

- KPI cards
- tables
- time series
- group comparisons
- ABC matrices

## Что уже можно считать legacy

- `analyze_products.py` в текущем виде

Причина:

- он использует накопительные продажи без временного окна;
- его выводы могут быть операционно неверными.

Его не обязательно удалять, но нужно явно считать `legacy / historical heuristic`.

## Нужно ли пересобирать проект?

Ответ:

- `нет`, если речь про полный rewrite;
- `да`, если речь про архитектурную сборку вокруг слоёв.

То есть правильный путь:

1. не ломать рабочие исследовательские скрипты;
2. постепенно вынести общую логику в ядро;
3. объявить, какие скрипты стратегические, какие операционные, какие legacy;
4. готовить JSON-выходы для будущего браузерного интерфейса.

## Самые правильные следующие шаги

1. Сделать официальный периодический анализ на базе `SELLS_REPORT` и `LEFT_OUT_REPORT`.
2. Отметить `analyze_products.py` как legacy.
3. Ввести папку `data/` и нормализованные датасеты.
4. Вынести общие CSV/HTTP helpers в общий модуль.
5. Добавить smoke tests на sample reports.
6. После этого уже делать интерактивный web UI.
