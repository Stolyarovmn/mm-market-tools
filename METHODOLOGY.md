# Methodology

Проект нужно делить на 2 уровня аналитики:

1. `Исторический слой`
   Что уже было накоплено по продажам, ассортименту и конкурентам.
2. `Оконный слой`
   Что реально продавалось за последние `7/14/30` дней.

Без второго слоя любой исторический хит может ошибочно выглядеть как актуальный бестселлер.

## Почему старые отчёты ограничены

Текущие CSV и часть ранних отчётов используют накопительный показатель `quantitySold`.

Это значит:

- товар мог продаться давно и сейчас почти не двигаться;
- товар мог быть временно недоступен, но всё равно оставаться "хитом" по сумме;
- старые товары получают системное преимущество перед новыми.

Поэтому накопительные отчёты подходят для:

- поиска исторических хитов;
- поиска сильных товарных идей;
- оценки ширины ассортимента.

Но они не подходят как единственная база для:

- заказа;
- прогноза остатков;
- оценки текущего спроса.

Отдельная проблема:

- сравнение только по названию товара слишком слабое;
- в карточке может быть несколько SKU или вариаций;
- в `LEFT_OUT_REPORT` одна карточка может быть разложена на несколько строк с разными `ШК`;
- поэтому title-only логика годится только как rough heuristic, но не как база для закупки и управленческих решений.

Identity-слой проекта должен идти в таком порядке:

1. `product_id` как семейство/карточка
2. `ШК` как конкретный вариант
3. `Seller SKU ID`
4. `SKU`
5. title только как fallback

## Что считать правильно

Для текущего спроса нужен анализ по окнам времени:

- `7 дней`
- `14 дней`
- `30 дней`

Источник в проекте для этого:

- `seller-analytics CubeJS`
- `documents reports` (`SELLS_REPORT`, `LEFT_OUT_REPORT`)
- регулярные снапшоты магазина `snapshot_shop.py` как fallback
- сравнение двух снапшотов через `analyze_time_window.py`

А для отдельного вопроса "что продаётся в рынке и по какой цене" нужен ещё один слой:

- рыночный/category scan по конкурентам
- группировка по товарным идеям
- price corridor по группе
- idea clusters внутри группы

## Приоритет источников

Для анализа по времени источники нужно использовать в таком порядке:

1. `CubeJS analytics`
2. `Documents reports`
3. `Shop snapshots`
4. Накопительные исторические выгрузки

Почему:

- `CubeJS` и `Documents reports` уже опираются на реальные окна времени;
- `Snapshots` это рабочий резервный путь, когда прямой доступ к аналитике нестабилен;
- накопительные числа нужны только для стратегического слоя и истории.

Важно:

- в реальной эксплуатации `Documents reports` могут вести себя нестабильно на шаге `create`;
- если в кабинете уже есть готовые выгрузки с теми же параметрами окна и магазина, их допустимо и полезно переиспользовать;
- это лучше, чем падать с ошибкой и терять весь weekly cycle.
- новый token нельзя считать рабочим "на глаз";
- его нужно прогонять через smoke-матрицу всех точек взаимодействия.

## Проверка нового token

При каждом новом token нужно проверять:

1. `CubeJS meta`
2. `CubeJS load`
3. `documents/requests`
4. `documents/create`

Практически это автоматизировано в:

- `validate_token_integrations.py`

Это важно, потому что разные точки могут вести себя по-разному:

- token может открывать `CubeJS`, но не позволять `documents/create`;
- token может быть рабочим для `documents/requests`, но часть CubeJS measures может давать `400`.

## Базовые общепринятые метрики

### 1. Sales Velocity

Формула:

`Sales Velocity = Units Sold in Window / Days in Window`

Смысл:

- показывает реальную скорость продаж сейчас;
- это базовая метрика для дозакупки и приоритизации SKU.

### 2. Stock Cover

Формула:

`Stock Cover Days = Current Stock / Average Daily Sales`

Смысл:

- показывает, на сколько дней хватит остатка;
- удобно для маркировки риска out-of-stock.

### 3. Sell-Through Rate

Классическая формула требует поставок и стартового остатка.
В проекте без приходов пока можно использовать прокси:

`Sell-Through Proxy = Sold in Window / (Sold in Window + Ending Stock)`

Смысл:

- помогает понять, что реально движется, а что лежит.

### 4. ABC Analysis

Классика:

- `A` — товары, дающие основную долю продаж;
- `B` — средний слой;
- `C` — длинный хвост.

Практика:

- заказывать и контролировать в первую очередь `A`;
- оптимизировать ассортимент в `C`.

ABC корректно считать именно по обороту или продажам за окно времени, а не по общему накопительному счётчику.

### 5. XYZ Analysis

Требует рядов по периодам.

Смысл:

- `X` — стабильный спрос;
- `Y` — сезонный / волнообразный;
- `Z` — нерегулярный.

Это особенно полезно после накопления серии снапшотов.

### 6. Price Ladder / Price Corridor

Анализ не одного товара, а идеи:

- в каких диапазонах цен товарная идея продаётся лучше;
- где у вас дыра в матрице.

Для вашего магазина это уже видно:

- основной рабочий диапазон `199-499 ₽`;
- `500-799 ₽` стоит расширять выборочно;
- `800+ ₽` только для подтверждённых идей.

### 7. Assortment Breadth / Depth

- `Breadth` — сколько товарных идей вообще покрыто;
- `Depth` — насколько глубоко развита каждая идея.

Для роста важно не просто добавлять SKU, а расширять идеи, где уже есть сигнал спроса.

### 8. Competitor Idea Benchmark

Сравнивать не только точные SKU, а:

- товарную идею;
- цену;
- силу продаж;
- ширину линейки;
- насыщенность группы.

Именно это делает `analyze_product_ideas.py`.

Но для вопроса рынка в более прямой форме теперь добавлен и другой модуль:

- `analyze_competitor_market.py`

Он отвечает не "какой конкурент похож на вас по идее", а "что вообще продаётся в категории, у кого и по какой цене".

Важно:

- `price_bands` и `idea_clusters` в рыночном модуле это эвристический слой для чтения рынка;
- это полезно для поиска направлений роста и ценовых коридоров;
- но это не замена точной SKU-нормализации внутри вашего магазина.

### 9. Period-over-Period Growth

Формула:

`Growth % = (Current Period - Reference Period) / Reference Period * 100`

Нужно считать как минимум для:

- trailing `365` days vs previous year same window
- current `YTD` vs previous `YTD`
- если доступно, same window `3` years ago

Это реализовано в:

- `cubejs_period_compare.py`

## Как использовать проект правильно

### Для стратегии

Использовать:

- `analyze_product_ideas.py`
- `build_growth_plan.py`
- `analyze_competitor_market.py`

### Для текущего спроса и заказов

Использовать:

- `cubejs_query.py`
- `request_document_report.py`
- `analyze_official_reports.py`
- `analyze_variant_families.py`
- `snapshot_shop.py`
- `analyze_time_window.py`

Важно:

- historical growth modules не должны напрямую рекомендовать дозакупку без проверки периода;
- закупка и replenishment должны опираться на seller reports / CubeJS / snapshots, а не на старые cumulative sales.
- если карточка содержит несколько вариантов, спорные решения нужно дополнительно смотреть на уровне семейства товара, а не отдельной строки отчёта.

## Рекомендуемый процесс

1. Снимать снапшот магазина каждый день или хотя бы каждые `2-3` дня.
2. Раз в неделю строить `7-day` анализ.
3. Раз в месяц строить `30-day` анализ.
4. Параллельно обновлять стратегический слой:
   - группы товаров;
   - идеи;
   - сравнение с конкурентами;
   - план расширения.
5. Складывать результат не только в markdown/csv, но и в:
   - `data/normalized`
   - `data/dashboard`

Это нужно, чтобы потом собрать браузерный интерфейс без переписывания аналитики.

## Что важно помнить

- Исторический хит не равен текущему хиту.
- Out-of-stock нельзя оценивать только по накопительным продажам.
- Мёртвый склад нельзя определять без окна времени.
- Конкурентов лучше сравнивать по идеям, а не только по точному совпадению карточек.
