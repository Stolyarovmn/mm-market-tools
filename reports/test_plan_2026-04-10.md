### 4.1 Base load

Шаги:

1. Запустить:

2. Открыть:
   - `http://127.0.0.1:8000/ui/`

Ожидание:

- selector заполнен;
- виден dashboard по latest report.

### 4.2 Theme toggle

Шаги:

1. Переключить тему.
2. Перезагрузить страницу.

Ожидание:

- тема меняется;
- выбор темы сохраняется.

### 4.3 Report switching

Проверить:

- `weekly_operational`
- `official_period_analysis`
- `cubejs_period_compare`
- `competitor_market_analysis`

Ожидание:

- metadata/kpi/panels меняются без JS ошибок;

### 4.4 Priority cards

Ожидание:

### 4.6 Action Center without runner

## 6. Online MM Workflow Cases

Эти кейсы требуют пользователя, потому что online MM refresh выполняется из отдельной рабочей сетевой сессии.

### 6.1 Token validation

Пользователь:

1. Запускает refresh runner.
2. Через браузер запускает `validate_token` из подходящей сетевой сессии.

Ожидание:

- статус `succeeded`;
- в log видны проверки endpoint’ов;
- после этого можно возвращаться к обычному циклу работы.

### 6.2 Weekly refresh

Пользователь:

1. Запускает `weekly_operational` из подходящей сетевой сессии.

Ожидание:

- raw CSV скачиваются;
- dashboard bundle обновляется;
- позже main UI видит новый report.

### 6.3 Market scan

Пользователь:

1. Запускает `market_scan` из подходящей сетевой сессии.

Ожидание:

- создаются `reports/*` и `data/dashboard/*`;
- main UI потом видит новый market report.

## 8. Manual QA Tasks To Ask User

Когда пользователь проснётся, можно делегировать:

1. Запустить `refresh UI` и прислать текст блока `Текущий запуск`.
2. Нажать:
   - `В список`
   - `Задача`
   и прислать текст `Action Center`.
3. Переключить 3-4 отчёта и прислать:
   - `Приоритеты Сейчас`
   - `Что Изменилось`
4. Из подходящей сетевой сессии запустить:
   - `validate_token`
   - `weekly_operational`
   и прислать лог/статус.
