# Research: Накладные, Себестоимость И Контракт Documents Create

Дата: `2026-04-13`

## Что удалось подтвердить

### 1. Накладные действительно являются отдельным рабочим контуром ЛК

По официальной инструкции продавца:

- в разделе `5.2` есть `Создание накладной для FBO товара`;
- в разделе `5.3` описана подготовка товара к отгрузке;
- на этапах отгрузки используются:
  - накладная;
  - лист отгрузки;
  - этикетки;
  - штрихкоды;
  - состав поставки.

Практический смысл для проекта:

- накладная — это не просто логистический документ;
- это естественный кандидат на исторический источник партий;
- через неё можно пытаться восстанавливать историю завоза и более точную себестоимость по времени.

### 2. Документация подтверждает отдельный финансовый контур

По `2.7 Отчеты и закрывающие документы` и `2.6 Вывод денежных средств` видно, что в ЛК существует не только sales analytics, но и отдельный money/document layer:

- закрывающие документы;
- вывод денежных средств;
- услуги и тарифы;
- платное хранение.

Это усиливает вывод, что прибыльный слой надо строить не только на продажах, но и на документах / услугах / storage.

### 3. `documents/create` сейчас не принимает даже безопасные варианты payload

Проверены варианты для текущего аккаунта:

- `SELLS_REPORT` с `MS_EXCEL`, `group=false`, `returns=false`, `shopIds=[98]`
- `SELLS_REPORT` с `MS_EXCEL`, `group=true`
- `SELLS_REPORT` с `CSV`, `group=false`
- `SELLS_REPORT` с `CSV`, `group=true`
- `SELLS_REPORT` через `shopId=98`
- `LEFT_OUT_REPORT` с `MS_EXCEL`

Результат по всем вариантам:

- `400 Validation failed`

Вывод:

- проблема не в одном поле `group` или `contentType`;
- текущий create-contract seller documents для этого кабинета либо изменился, либо требует скрытые параметры, которых нет в нашей текущей реализации;
- пока надёжная стратегия — опираться на `documents/requests` и reuse уже существующих completed отчётов, а не рассчитывать на create как на стабильный контракт.

## Что это значит для проекта

### По накладным

Следующий ценный слой — это ingestion накладных как источника:

- партии;
- даты завоза;
- потенциальной себестоимости по времени;
- связи между закупкой, продажей, хранением и остатком.

### По `SELLS_REPORT`

Нужно менять подход:

1. не ждать, что `documents/create` стабильно заработает сам по себе;
2. сначала научиться максимизировать reuse исторически доступных completed report'ов;
3. отдельно искать, существует ли другой актуальный contract или endpoint для нового generation flow.

## Рекомендуемый execution order

1. Поднять отдельный research/ingestion контур по накладным.
2. Для operational pipeline временно держать стратегию `reuse-first`.
3. Только потом снова возвращаться к reverse-engineering `documents/create`.
4. После этого чинить parser `PAID_STORAGE_REPORT`.
5. Затем брать `PaidPromo*`.

## Практический вывод

Если цель — рост прибыли, то накладные важнее, чем продолжать бесконечно биться в `documents/create`.

Причина простая:

- даже идеальный fresh `SELLS_REPORT` без нормального cost layer всё равно оставляет слепую зону по реальной экономике;
- накладные потенциально закрывают именно эту слепую зону.
