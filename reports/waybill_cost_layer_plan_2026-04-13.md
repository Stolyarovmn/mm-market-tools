# Implementation Plan: Waybill Cost Layer

Дата: `2026-04-13`

## Цель

Превратить накладные в отдельный исторический источник себестоимости и партий, чтобы проект умел считать прибыль не только по текущим manual override, но и по фактической истории завоза.

## Что уже подтверждено

Из официальной инструкции по созданию накладной:

- накладная может создаваться из Excel-файла;
- на листе `Товары на отправку` продавец указывает:
  - штрихкод;
  - себестоимость;
  - количество;
- затем выбирается таймслот поставки;
- для части товаров указывается идентификатор УПД;
- накладная живёт в отдельном разделе ЛК `Накладные`.

Это означает, что накладная уже содержит минимально достаточный набор для cost layer:

- идентификатор товара через штрихкод;
- unit cost;
- batch quantity;
- дату/момент поставки.

## Какой слой хотим получить

### 1. Сущность `waybill_batch`

Минимальные поля:

- `waybill_id`
- `created_at`
- `planned_supply_at`
- `accepted_at` если доступно
- `barcode`
- `sku`
- `product_id` если получится связать
- `title`
- `quantity_supplied`
- `unit_cogs`
- `batch_cogs_total`
- `upd_id` если есть
- `source_file`
- `source_row`

### 2. Нормализованный слой `historical_cogs`

Что должен уметь:

- хранить историю себестоимости по SKU / barcode;
- выдавать:
  - последнюю известную себестоимость;
  - медианную;
  - последнюю по времени до выбранного окна продаж;
  - диапазон изменения себестоимости по времени.

### 3. Связка с продажами

Нужны режимы расчёта:

- `latest_before_sale`
  Берём последнюю известную себестоимость до даты продажи.
- `latest_known`
  Берём последнюю известную себестоимость вообще.
- `manual_override_priority`
  Если есть ручной override, он сильнее batch-cost.
- `batch_history_reference`
  Показываем историю партий и не скрываем разброс цены закупки.

## Как это использовать для прибыли

### 1. Более честная маржа

Сейчас у проекта есть cost layer, но он частично держится на:

- полях из продаж;
- ручных CSV override;
- незаполненных blind spots.

Накладные дадут:

- историческую цену закупки;
- меньше слепых зон по cost coverage;
- более честный profit per SKU.

### 2. Решения по распродаже

Если связать:

- себестоимость из накладной;
- остаток;
- хранение;
- продажи;

то можно выделять SKU, которые:

- лежат давно;
- покупались дорого;
- продаются медленно;
- ещё и тянут хранение.

Это прямой список на:

- срочную распродажу;
- отказ от дозакупки;
- вывод из ассортимента.

### 3. Решения по перезаказу

Если товар:

- продаётся быстро;
- storage cost низкий;
- маржа по batch-cost хорошая;

то это уже кандидат на безопасное усиление закупки.

## Реализация по шагам

### Шаг 1. Research contract

Найти, как именно получать накладные:

- экспортом из ЛК;
- через seller API, если такой endpoint существует;
- через загрузку локального Excel, если продавец скачивает накладные вручную.

ToD:

- подтверждён минимум один воспроизводимый путь получения файла накладной.

Тест:

- есть хотя бы один реальный файл накладной или подтверждённый endpoint/экспортный flow.

### Шаг 2. Parser waybill Excel

Сделать parser для накладной.

Что извлекаем в первую очередь:

- штрихкод;
- себестоимость;
- количество;
- дата/таймслот;
- идентификатор УПД;
- номер/идентификатор накладной.

ToD:

- parser стабильно разворачивает хотя бы один реальный файл в rows `waybill_batch`.

Тест:

- на выходе есть JSON/CSV с batch rows;
- значения `barcode`, `unit_cogs`, `quantity_supplied` не пустые хотя бы у части строк.

### Шаг 3. Identity matching

Связать batch rows с существующим entity layer.

Приоритет ключей:

1. `barcode`
2. `sku`
3. `product_id`
4. title fallback только как слабый режим

ToD:

- значимая доля строк связывается с нашим SKU/entity слоем.

Тест:

- match rate измеряется числом и процентом;
- unmatched rows сохраняются отдельно.

### Шаг 4. Historical COGS store

Собрать persistent store по партиям и исторической себестоимости.

ToD:

- можно по SKU получить историю batch-cost по времени.

Тест:

- для тестового SKU видно несколько cost points или хотя бы один надёжный batch-cost.

### Шаг 5. Merge с operational layer

Связать накладные с:

- продажами;
- остатками;
- storage cost.

ToD:

- operational bundle умеет показывать не только текущую себестоимость, но и источник `waybill/manual/sales-derived`.

Тест:

- для SKU видно origin cost;
- можно считать profit с учётом batch-cost.

### Шаг 6. Manager-facing dashboard

Добавить отдельный слой:

- `История себестоимости`
- `Партии и закупки`
- `SKU с дорогой закупкой и слабой продажей`
- `SKU с хорошей маржой и быстрым движением`

ToD:

- в dashboard появился новый cost/history слой, который реально помогает принимать решения.

Тест:

- на небольшом подмножестве SKU можно увидеть, какие позиции выгодно распродавать, а какие усиливать.

## Что не делать сейчас

- не уходить в FBS;
- не усложнять модель до FIFO/LIFO, пока нет базового batch layer;
- не строить auto-repricing от batch-cost до подтверждения качества matching.

## Ближайший следующий шаг

Нужен один реальный образец накладной или подтверждённый способ его выгрузки. После этого можно сразу делать parser и поднимать первый рабочий `waybill_cost_layer`.
