## Цель

Зафиксировать по пунктам:

- какую информацию можно получить из личного кабинета продавца Магнит Маркета;

## Короткий вывод

Личный кабинет даёт существенно больше данных, чем проект использует сейчас.

Уже реально забираем:

- продажи и часть экономики через `SELLS_REPORT`;
- остатки / OOS / идентификацию SKU через `LEFT_OUT_REPORT`;
- аналитический слой `Sales` через CubeJS;
- слой причин возвратов через CubeJS `SalesReturn`;
- слой платного хранения через `PAID_STORAGE_REPORT` как offline/source layer.

Пока не забираем или забираем не полностью:

- ежедневные списания и детализацию услуг из раздела `Финансы`;
- накладные как отдельный источник себестоимости и истории поставок;
- закрывающие документы;
- рекламную аналитику `PaidPromo*`;
- weekly email report;
- товары с истекающим сроком годности;
- transactional seller API по товарам / ценам / остаткам;
- отзывы и чат;
- маркетинговые сущности ЛК: промокоды, таймерные акции, буст.

## Инвентарь по разделам ЛК

### 1. Главная / сводки магазина

Что можно получить:

- базовые сводки по магазину;
- операционные показатели по продажам;
- финансовые показатели;
- переходы в аналитику, финансы, товары, маркетинг.

Что уже получаем:

- частично;
- manager-facing сводки собираются через:
  - `weekly_operational_report.py`
  - `cubejs_period_compare.py`
  - market/pricing/content layers

Что не закрыто:

- прямого cabinet-home ingestion слоя нет;
- нет сверки home-summary ЛК против наших bundle.

### 2. Аналитика

Что можно получить:

- продажи;
- заказы;
- средний чек;
- уникальных покупателей;
- выручку;
- прибыль / маржу;
- возвраты и причины возвратов;
- разрезы по датам;
- разрезы по категориям;
- разрезы по товарам и SKU;
- закупочную цену и производные экономические показатели.

Что подтверждено live:

- в `CubeJS / Sales` доступны:
  - `seller_revenue_measure`
  - `seller_profit`
  - `seller_margin_measure`
  - `avg_receipt_measure`
  - `unique_buyers`
  - `orders_number`
  - `items_purchase_price`
  - `seller_purchase_price_sum`
  - `order_item_seller_purchase_price`
    - `category_id`
  - `product_id`
  - `sku_id`
- в `CubeJS / SalesReturn` доступны:
  - `returned_quantity_measure`
  - `number_of_returns_by_cause`
  - `cause`
  - `amount_to_return`
  - `shop_id`
  - `product_id`
  - `sku_id`
  
Что уже получаем:

- да, и это один из самых сильных покрытых слоёв проекта;
- используются:
  - `cubejs_period_compare.py`
  - `cubejs_query.py`
  - `build_sales_return_report.py`

Что не закрыто:

- не используем весь объём мер `Sales`;
- нет отдельного слоя по покупателям, AOV, FBS/FBO mix и profitability tree.

### 6. Накладные и поставки

Что можно получить:

- акты на отгрузку / накладные;
- поставки на склад;
- состав поставки;
- историю поставок;
- связанный контур себестоимости, если она вносится через накладные;
- возвратные акты и движения товара со склада.

Почему это важно:

- по словам владельца магазина, именно в накладных видна себестоимость;
- это может дать более правдивый и историчный cost layer, чем ручные CSV override.

Что уже получаем:

- напрямую из накладных не получаем;
- косвенно cost-поля уже видны в:
  - `SELLS_REPORT`
  - `CubeJS / Sales`
  - локальном `zero_cogs` / manual override контуре

Что не закрыто:

- нет отдельного парсера накладных;
- нет истории изменения себестоимости по партиям;
- нет reconciliation между закупкой, продажей и остатком.

Статус:

- `не используется как отдельный источник`

## Самые ценные gaps сейчас

1. Накладные как исторический источник себестоимости и партий.
2. Paid promo analytics из `PaidPromo*`.
3. Финансовый слой `доступно к выводу / в обработке / возвраты / услуги`.
4. Transactional seller API по товарам, ценам, остаткам и FBS.
5. Отзывы как источник причин брака, возвратов и проблем карточки.

## Практический вывод для проекта

Если смотреть именно на ценность для магазина, следующий приоритет такой:

1. Подтянуть накладные и связать их с cost layer.
2. Добить live `PAID_STORAGE_REPORT` и собрать единый cost-of-storage слой.
3. Поднять paid promo dashboard из `PaidPromo*`.
4. Сделать seller transactional sync:
   - товары;
   - цены;
   - остатки;
   - FBS.
5. После этого брать отзывы и чат как customer-feedback layer.
